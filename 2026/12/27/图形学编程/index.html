图像位数与像素

🎯 核心：位数 = 调色盘大小

一个像素就是一个格子，只能涂一种颜色。
位数决定“这种颜色可以从多少种里选”。

📊 位数对比表

位数 每像素字节 颜色数量 调色盘大小 效果
8位 1字节 256色 256格 可见色块、渐变色带
16位 2字节 65536色 6.5万格 老游戏，轻微色带
24位 3字节 1677万色 1677万格 真彩色，平滑渐变
32位 4字节 1677万色+透明度 1677万格+Alpha 真彩色+半透明

📏 位数本质：切分亮度

把0-100%的亮度切成多少份：

位数 份数 台阶粗细 用途
1位 2份 纯黑/纯白 黑白图
8位 256份 肉眼极限 JPG、网页
10位 1024份 专业显示器 HDR
12位 4096份 RAW格式 摄影后期
16位 65536份 无损编辑 PS修图

24位 = 每个通道8位 = 256级亮度 × 3个通道
不是1677万种独立的颜色，是256级红 × 256级绿 × 256级蓝的组合。

💎 一句话精髓

位数不是决定“涂几次颜色”，是决定“调色盘里有多少格颜色可选”。
24位的1677万色不是物理存在，是256×256×256的数学组合。

BMP原点与高度字段

BMP文件存储本质
BMP文件里只有二进制，没有“负号”
文件偏移 存储的4字节 解释为int 含义
22-25 02 00 00 00 +2 高度2，从下往上存
22-25 FE FF FF FF -2 高度2，从上往下存
负号不存在文件里，是你用有符号int读出来的。
两个过程，同一字段
1️⃣ 存文件时（别人写BMP）
高度值 像素顺序 文件第一行是
正数 下→上 底部
负数 上→下 顶部
2️⃣ 读文件时（你写loadBMP）
读到的高度 含义 你需要
正数 文件是下→上 翻转（倒序）
负数 文件是上→下 不翻转（直接放）

PNG

PNG是什么？

PNG = Portable Network Graphics = 可移植网络图形格式
✅ 无损压缩（解压后和原来一模一样！）
✅ 支持透明通道（Alpha）
✅ 支持真彩色、索引色、灰度
✅ 比GIF好10倍，还免费！
❌ 比JPG大（照片不用它）
❌ 解码比JPG快，比BMP慢
❌ 不支持CMYK（印刷用不了）

​

PNG的核心哲学

2.1 PNG vs GIF 恩怨情仇
1995年，Unisys公司宣布：GIF用的LZW压缩我们专利了，要收费！
互联网炸了 → PNG诞生：我们要做免费的GIF替代品！

所以PNG天生：
✅ 免费！免费！免费！
✅ 比GIF压缩率更高
✅ 支持更多颜色（GIF只有256色）
✅ 支持真Alpha透明（GIF只有1位透明）
✅ 专利友好

​
2.2 PNG vs JPG 定位不同
PNG: 无损、透明、边缘锐利 → 适合图标、UI、截图、文字
JPG: 有损、照片级 → 适合风景、人像、渐变

经典应用：

  • 你的地球仪纹理 → PNG (透明背景)
  • 游戏UI图标 → PNG
  • 微信截图 → PNG
  • 朋友圈自拍 → JPG

第1步：滤波器（核心中的核心）

6.1 为什么需要滤波器？
// 原始像素（RGBA）：
[255,0,0,255] [255,1,0,255] [254,2,0,255] [255,3,0,255]
红 微红 微暗红 更暗红

// 直接压缩：数据差异大，不好压
// 但相邻像素很接近啊！

// 滤波器 = 存"差值"而不是"原值"
[255,0,0,255] [+1,0,0,0] [-1,0,0,0] [+1,0,0,0]
基准值 差值 差值 差值

// 数字变小了 → 哈夫曼编码更短！

​
6.2 PNG的5种滤波器（每行选一个）
// 0. None: 不滤波
Filt(x) = Orig(x)

// 1. Sub: 减去左边像素
Filt(x) = Orig(x) - Orig(x-1)

// 2. Up: 减上边像素
Filt(x) = Orig(x) - Orig(x - rowBytes)

// 3. Average: 减(左+上)/2
Filt(x) = Orig(x) - (Orig(x-1) + Orig(x - rowBytes)) / 2

// 4. Paeth: 减Paeth预测值（最聪明）
Filt(x) = Orig(x) - Paeth(左, 上, 左上)

第2步：DEFLATE压缩

7.1 DEFLATE = LZ77 + 哈夫曼
LZ77: 找重复字符串，用(距离,长度)替换
“ABABAB” → “AB(4,3)” // 从4个字符前复制3个

哈夫曼: 高频字符用短编码，低频用长编码
A出现100次 → 编码"0"
B出现50次 → 编码"10"
C出现1次 → 编码"110"

​
7.2 zlib库（PNG实际用的）
// PNG不是直接存DEFLATE数据，外面还包了zlib头
[zlib头:2] [DEFLATE数据…] [zlib校验:4]

zlib头 = 压缩方法(8) + 窗口大小 + 校验位

​
7.3 为什么PNG比GIF小？
GIF: LZW压缩（受专利限制，优化一般）
PNG: DEFLATE（LZ77+哈夫曼，优化到极致）

相同图片，PNG比GIF小20-30%！

PNG解码流程（STB在做什么）

  1. 读8字节签名 → 确认是PNG

  2. 循环读数据块：

    • 读长度、类型
    • 读数据、CRC
    • 验证CRC（防止文件损坏）

  3. 处理IHDR块 → 宽、高、位深、颜色类型

  4. 如果有PLTE → 读调色板

  5. 收集所有IDAT块 → 合并成完整压缩流

  6. zlib解压 → 得到滤波后的原始数据

  7. 逐行逆滤波：

    • 读每行第1字节 → 滤波器类型
    • 应用反向滤波器 → 恢复真实像素

  8. 如果是亚当7 → 重新排列像素顺序

  9. 颜色转换：

    • 索引色 → 查调色板
    • 灰度 → RGB
    • YCbCr? 不用，PNG是RGB

  10. 输出RGBA/RGB数组

着色器相关OpenGL函数

背景知识

  • 这些 gl* 函数是 OpenGL API，不是 glad 的函数
  • glad 的作用：运行时从显卡驱动拿到这些函数的地址
  • 你调 glCreateShader，实际执行的是显卡驱动里的代码
  • glad.h 里声明了函数指针，glad.c 里定义并初始化它们

关于 gl.h 和 glu.h

  • gl.h：OpenGL 1.1 的核心头文件（1997年冻结）
  • glu.h：GLU工具库头文件（同样1997年冻结）
  • 现代OpenGL用glad生成的glad.h替代它们
  • gluPerspective这类函数来自glu.h，不是OpenGL核心

OpenGL函数加载机制

  • 显卡驱动包含OpenGL 1.1-4.6的所有实现
  • 系统opengl32.dll只有1.1的函数入口
  • 新函数（2.0以后）需要运行时从驱动获取地址
  • glad/glew负责这个"获取地址"的工作
